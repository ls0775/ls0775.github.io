Observability: Prometheus, Grafana, and the First SLOs

9 May 2026

Tags: homelab, prometheus, grafana, observability, sre, slo, kubernetes

Up until this point, I knew the cluster was running because I could run kubectl get pods and see green. That's not observability — that's checking your pulse with your fingers. This week is about deploying real monitoring and doing something I didn't expect: writing SLO definitions for a homelab.

What You Get Out of the Box

The kube-prometheus-stack Helm chart is one of the more impressive pieces of Kubernetes tooling I've used. A single helm install deploys:

  • Prometheus — metrics collection and storage
  • Grafana — dashboards and visualization
  • Alertmanager — alert routing and deduplication
  • Node Exporter — hardware-level metrics (CPU, memory, disk, network) on every node
  • kube-state-metrics — Kubernetes object state (is this Deployment at the desired replica count? are Pods in CrashLoopBackoff?)
  • Prometheus Operator — manages Prometheus configuration via Kubernetes CRDs

The chart is deployed into the monitoring namespace. After deployment, Prometheus is scraping 17 targets every 30 seconds, and Grafana comes pre-loaded with 8+ dashboards covering node health, pod health, CPU/memory, and Prometheus internals.

The first time I opened Grafana at https://grafana.lab.ls0775.com and saw real-time graphs of my cluster's CPU usage, NFS I/O, and pod restart counts, it felt like putting on glasses. I could suddenly see the cluster.

What the Stack Scrapes

Prometheus collects metrics from:

  • The Kubernetes API server (apiserver) — request latency, error rates, etcd operation timing
  • kubelet on each node — pod lifecycle, container resource consumption
  • node-exporter — physical hardware metrics (disk I/O, network throughput, load average, memory pressure)
  • kube-state-metrics — object-level Kubernetes state (replica counts, job completions, pod phase)
  • coredns — DNS query rates and latency
  • alertmanager and Prometheus itself — meta-monitoring (is the monitoring system healthy?)

The scrape target auto-discovery is handled by the Prometheus Operator via ServiceMonitor objects. The key configuration I set:

prometheus:
  prometheusSpec:
    serviceMonitorSelectorNilUsesHelmValues: false

This makes Prometheus discover all ServiceMonitor objects in the cluster, not just ones matching the chart's own labels. When I add a new application and want it to expose metrics, I create a ServiceMonitor and Prometheus picks it up automatically — no Prometheus configuration change required.

Writing SLOs for a Homelab

Here's where this project shifted from "homelab" to "production learning lab" in a concrete way: I defined Service Level Objectives.

An SLO is a target for reliability: "This service should be available 99.9% of the time over a 30-day rolling window." The measurement of whether you're meeting it is a Service Level Indicator — a specific Prometheus query that produces a number between 0 and 1.

For the homelab, I defined four:

Service What We Measure Target
Pi-hole DNS Query success rate 99.9%
k3s API server API availability (up metric) 99.5%
NFS storage Mount availability per node 99.9%
Argo CD UI Probe success via Blackbox Exporter 99.0%

Writing these forced me to answer "what does 'working' mean for this service?" — which is a much harder question than it looks. For Pi-hole DNS, "working" means DNS queries are being resolved, not just that the Pod is Running. For the k3s API, "working" means the API is accepting requests, not just that the node is Up.

The SLO definitions are stored in docs/operations/slo-definitions.md. They include the Prometheus queries alongside the targets.

Error Budgets: The Math Behind "Is This Downtime OK?"

An SLO of 99.5% means 0.5% of a 30-day window (approximately 3.6 hours) is the "error budget" — the amount of unavailability you've agreed to tolerate per month.

If the k3s API is down for 2 hours in a month, you've spent about 55% of your error budget. If it goes down again for another 2 hours the same month, you've exhausted the budget and any further downtime is "above SLO."

In a real SRE environment, this matters because it gates feature releases (if the error budget is depleted, new features that add risk are blocked until it refills) and it triggers post-mortems (budget exhaustion requires a written incident review).

In a homelab, the consequence is less severe — I'm not blocking my own releases. But tracking error budgets trains the habit of asking "what's the impact of this downtime?" rather than just "is it back up yet?" Those are very different questions, and the second one is how on-call engineers think.

The Honest Gap: Loki Is Not Deployed

I want to be explicit about something: Prometheus tells you what is happening. It does not tell you what happened.

If a pod crashed at 2am and recovered by the time I wake up, Prometheus will show a spike in container restart counts — but I can't read the logs from the crashed container. The container is gone, and without a log aggregation system, those logs are gone with it.

Loki (paired with Promtail, which ships logs from pods to Loki) is the missing piece. It would let me query historical pod logs in Grafana, enabling real post-mortem analysis: "The container crashed — why? What was it logging in the 30 seconds before the crash?"

Loki is on the Phase 4 backlog (risk register: R-004, High severity). It's not deployed yet, and I'd be misrepresenting the setup if I didn't note that. The monitoring stack is genuinely useful without Loki — Prometheus + Grafana covers the "what is happening" question well. But a production observability stack has logs, metrics, and traces. I have one of the three.

What Recording Rules Should Be (But Aren't Yet)

The Prometheus queries in my SLO definitions run on-demand when a dashboard loads. The right approach is to precompute them as Recording Rules — Prometheus evaluates the query on a schedule and stores the result as a new time series. This reduces dashboard query latency and enables alerting on the pre-computed values.

Implementing the Recording Rules is on the backlog (risk register: R-005, Medium). The SLO definitions exist; the automation of measuring them doesn't yet. Knowing the gap is progress.

What This Taught Me

SLO definitions force precision about what "working" means. The process of writing a Prometheus query that accurately captures "Pi-hole DNS is handling queries" is harder than it sounds — you have to understand the metrics the service exposes, what they mean, and what failure looks like in metric form. That precision carries over to how I think about system reliability generally.

The kube-prometheus-stack also taught me that "monitoring" and "observability" are different. Monitoring tells you when something is wrong. Observability tells you why. Prometheus + Grafana is monitoring. Logs + traces + metrics together is observability. I have the metrics layer; the full stack is still in progress.
