Migrating k3s from SQLite to etcd HA

25 April 2026

Tags: homelab, kubernetes, k3s, etcd, high-availability, database

A single-node k3s cluster stores all cluster state in a SQLite database on the server node. SQLite is fine for development — it's simple, zero-maintenance, and doesn't require a separate database process. It's not fine for HA. This post covers the migration to etcd and what "Kubernetes HA" actually means.

What's in the Kubernetes Database?

Before getting into the migration, it helps to understand what's being stored and why it matters.

The Kubernetes API server is stateless — every request handler reads from and writes to a backing store. That store is etcd (or in k3s's case, optionally SQLite). Every Pod spec, every ConfigMap, every Secret, every Deployment, every node registration — it all lives in the database. If the database is gone, the cluster state is gone.

With SQLite, that database is a single file on k3s-server-01. If k3s-server-01 goes down, you can't read or write cluster state. Pods continue running on worker nodes (kubelet is autonomous), but you can't deploy new things, scale existing workloads, or respond to failures. The control plane is effectively down.

With etcd, the database is replicated across multiple server nodes using the Raft consensus algorithm. As long as a majority of etcd members are alive, the cluster state is accessible and writable. With two server nodes, you lose HA at the etcd level (one failure means 1 of 2 members, which is not a majority — but k3s handles this with a specific two-node configuration). With three, you have true fault tolerance.

The Migration Process

k3s supports upgrading from SQLite to etcd by deploying a new server with --cluster-init (which initializes an etcd backend) and joining additional servers to it. I'd already deployed k3s-server-01 with --cluster-init, so it was already running etcd internally.

The migration step was joining k3s-server-02 as a second server. This is done by running k3s on server-02 with the --server flag pointing at server-01 and providing the cluster token:

# On k3s-server-02
K3S_TOKEN=$(ssh root@192.168.99.21 cat /var/lib/rancher/k3s/server/node-token)
curl -sfL https://get.k3s.io | sh -s - server \
  --server https://192.168.99.21:6443 \
  --token $K3S_TOKEN \
  --tls-san k3s-server-02.lab.ls0775.com \
  --tls-san k3s-cluster.lab.ls0775.com \
  --tls-san 192.168.99.22

After a minute or two, k3s-server-02 appeared in the node list:

kubectl get nodes
# k3s-server-01   Ready   control-plane,master
# k3s-server-02   Ready   control-plane,master

I then verified etcd replication was working:

kubectl -n kube-system exec etcd-k3s-server-01 -- \
  etcdctl --endpoints=localhost:2379 member list
# Should show both server nodes as etcd members

DNS Round-Robin for HA API Access

With two control-plane nodes, I needed the cluster API endpoint to be resilient to either node going down. The solution: DNS round-robin in Pi-hole.

I added two A records for k3s-cluster.lab.ls0775.com:

192.168.99.21  k3s-cluster.lab.ls0775.com
192.168.99.22  k3s-cluster.lab.ls0775.com

When my workstation connects to k3s-cluster.lab.ls0775.com, Pi-hole returns both IPs. kubectl tries the first, and if that fails (because the node is down), the OS retries with the next. It's not perfect load balancing, but for control-plane access from a workstation it's sufficient.

I updated the kubeconfig on my workstation to use k3s-cluster.lab.ls0775.com:6443 as the server address instead of a specific node IP.

The Honest Trade-Off: 2-Node etcd Is Still Fragile

Here's the part I want to be explicit about: a 2-node etcd cluster is not truly HA.

Raft consensus requires a majority quorum. With 2 nodes, a majority is 2. If either node goes down, you don't have 2 of 2 nodes — you have 1 of 2, which is not a majority. The cluster becomes read-only (or stops accepting writes, depending on the configuration).

k3s has a specific mode for 2-node clusters that's more tolerant than vanilla etcd's 2-node behavior, but the fundamental math doesn't change: losing one of two etcd members is bad.

The right fix is a third control-plane node. I have one planned (a small VM on the Synology VMM, tainted so it only runs control-plane workloads, not application workloads). It's in the risk register as R-001 (Critical). It's not deployed yet because adding a third node requires careful planning around the Synology's CPU budget, and I want to do it in Phase 5 properly rather than rush it now.

Documenting this gap explicitly is important. If I were showing this to an employer, a "we know this risk and here's the plan to address it" is more credible than a setup that looks HA but has a known fragility swept under the rug.

etcd Snapshots to Synology NFS

With etcd handling cluster state, I set up automated snapshots synced to the Synology NFS mount. k3s has built-in snapshot support:

# k3s etcd snapshot configuration in /etc/rancher/k3s/config.yaml
etcd-snapshot-schedule-cron: "0 */6 * * *"  # every 6 hours
etcd-snapshot-retention: 30                  # keep 30 snapshots
etcd-snapshot-dir: /var/lib/rancher/k3s/server/db/snapshots

A separate script syncs the snapshot directory to the Synology NFS mount at /volume1/proxmox/k3s-etcd-snapshots. This runs on a cron schedule and keeps the last 30 snapshots (roughly 7.5 days of history at 6-hour intervals).

What This Taught Me

The migration from SQLite to etcd made the Kubernetes control plane design tangible in a way that reading documentation didn't. etcd's Raft consensus, the quorum math, the difference between "cluster is running" and "cluster can make decisions" — these are concepts that I'd read about but not felt. Having to think through "what happens if this node goes down right now" for each stage of the migration made those concepts real.

The snapshot setup also drives home a point about distributed systems: replication is not backup. etcd replication protects against node failures. An etcd snapshot protects against data corruption or accidental deletion. You need both, and they serve different purposes.
