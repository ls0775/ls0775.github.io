GitOps: Deploying Applications with Argo CD

2 May 2026

Tags: homelab, gitops, argocd, kubernetes, helm, continuous-delivery

GitOps is one of those terms that gets used in job postings constantly and explained in blog posts abstractly. This week is my attempt at the concrete version: what GitOps actually means in day-to-day operation, and why I chose Argo CD to implement it.

What GitOps Actually Is

GitOps is a deployment model where git is the single source of truth for desired system state. You don't deploy by running kubectl apply on your laptop. You deploy by pushing a change to git, and a tool running in the cluster detects that change and applies it.

The key shift: your deployment pipeline runs inside the cluster, not outside it. The cluster knows what it's supposed to look like (the git repository), continuously compares that to what it actually looks like, and reconciles any differences.

This has a few concrete consequences:

Drift detection. If someone manually runs kubectl edit and changes a deployment, Argo CD detects that the live state has drifted from the git state and shows you the diff. You can choose to sync back to git (reverting the manual change) or update git to match reality.

Auditability. Every change to the cluster goes through git — with a commit hash, an author, and optionally a PR review. "Who changed that deployment config at 2am?" becomes a git blame question, not a cluster audit log archaeology project.

Self-service. Adding a new application doesn't require SSH access to the cluster. You add the Kubernetes manifests to the git repository, commit, and Argo CD picks it up.

Why Argo CD Over Flux

Both Argo CD and Flux implement GitOps for Kubernetes. I chose Argo CD for a few reasons:

The UI. Argo CD has a web interface that shows application sync status, resource trees, and diff views. Flux is CLI-first. For a homelab where I'm learning, the visual representation of "what's in the cluster and how is it structured" is genuinely educational. I've spent time just clicking around the Argo CD UI understanding how resources relate to each other.

RBAC model. Argo CD has a built-in RBAC system with projects and role mappings. I defined three roles: argocd-readonly (can view), argocd-deploy (can sync), argocd-admin (can manage Argo CD itself). Even in a solo homelab, modeling RBAC correctly is good practice.

App-of-apps pattern. Argo CD supports managing multiple applications through a single "meta-application" that points at a directory of application definitions. This makes adding new apps to the cluster as simple as adding a file to the repository.

Bootstrap: The One Manual Apply

There's an inherent chicken-and-egg problem with GitOps: the tool that applies things from git has to be applied manually the first time. Argo CD is the exception to the GitOps rule — you install it imperatively, and then use it to manage everything else declaratively.

The bootstrap sequence:

# 1. Apply the argocd namespace
kubectl apply -f k8s/apps/argocd/namespace.yaml

# 2. Install Argo CD via Helm
helm repo add argo https://argoproj.github.io/argo-helm
helm repo update
helm upgrade --install argocd argo/argo-cd \
  --namespace argocd \
  -f k8s/apps/argocd/helm-values.yaml

# 3. Apply the ingress resource
kubectl apply -f k8s/apps/argocd/ingress.yaml

After that, Argo CD is running at https://argocd.lab.ls0775.com and I get the initial admin password from a secret:

kubectl -n argocd get secret argocd-initial-admin-secret \
  -o jsonpath="{.data.password}" | base64 -d

From this point forward, I manage applications through Argo CD rather than direct kubectl apply. The Helm values that configure Argo CD itself live in the git repository at k8s/apps/argocd/helm-values.yaml — so Argo CD is, in a sense, managing its own configuration.

The Helm Values in Git

Rather than letting Helm manage Argo CD with default values, I have explicit values in k8s/apps/argocd/helm-values.yaml. This file is the configuration that matters:

  • Metrics enabled for Prometheus scraping on all components (controller, server, repo-server, applicationset)
  • RBAC policy set to role:readonly by default, with explicit role mappings
  • OIDC disabled (scaffold is there, but OIDC requires an IdP that I haven't deployed yet)
  • Ingress configured for argocd.lab.ls0775.com via Traefik, using the wildcard TLS cert

Having these values in git means a cluster rebuild is repeatable — I'm not relying on manual configuration that lives only in the running Argo CD instance.

Drift Detection in Practice

The most educational experience with Argo CD was seeing drift detection work on my first "oops." I'd manually patched a resource with kubectl edit to fix something quickly, intending to update the git file "later." Argo CD immediately showed the application as OutOfSync with a diff showing exactly what changed.

That visual cue — the red "OutOfSync" status in the UI — is a powerful prompt to keep git as the source of truth rather than letting manual changes accumulate. At work, this is enforced by policy. In a homelab, you have to enforce it on yourself, and Argo CD's UI makes that easier.

What This Taught Me

GitOps changes the mental model from "I deploy things" to "git declares things, the cluster converges to them." That shift is subtle but important. When you're thinking about deploying something new, the question becomes "what does the git state need to look like?" rather than "what commands do I need to run?" — and that reframe makes complex deployments more systematic.

The bootstrap problem (you have to manually apply the thing that applies everything else) is also a real operational consideration that most GitOps guides gloss over. In production, you'd automate the bootstrap with a CI/CD pipeline or a dedicated bootstrapping tool. For a homelab, doing it once manually is fine — but understanding that "this part is manual" prevents the false impression that GitOps is fully automated from day zero.
