Building a Proxmox HA Cluster on Two NUCs

11 April 2026

Tags: homelab, proxmox, high-availability, quorum, corosync

Building a "highly available" cluster is one of those things that looks simple until you try to do it with exactly two nodes. Two nodes is the worst-case scenario for distributed consensus, and there's a specific, non-obvious problem you hit immediately. This post is about that problem — and the solution I found.

Why Proxmox Over VMware ESXi

The obvious alternative to Proxmox for a homelab hypervisor is VMware ESXi. A lot of homelab guides use ESXi because you can get a free license for a single host, it's well-documented, and it's what you'll see in enterprise environments.

I went with Proxmox VE for three reasons:

It's open source. No license required, no waiting for VMware to decide what "free tier" means this year. Given Broadcom's acquisition of VMware and the subsequent licensing changes, this felt like the right call.

It's API-driven from day one. The bpg/proxmox Terraform/OpenTofu provider is mature and well-maintained. I wanted to provision VMs through code, not through a GUI wizard. Proxmox's API makes that straightforward.

LXC and KVM in the same interface. Proxmox supports both full VMs (KVM) and containers (LXC). I ended up using VMs for everything important (security boundaries matter), but having the option is useful.

The trade-off: Proxmox documentation can be inconsistent, and some of the web UI design choices are... opinionated. But for a learning environment where you want to understand what's happening underneath, transparent architecture beats polished UX.

The Two-Node Problem

Here's the thing nobody tells you until you're already committed: a two-node cluster doesn't actually give you high availability by default.

Corosync — the cluster communication layer that Proxmox uses — requires a quorum to make decisions. Quorum means "more than half the votes agree." With two nodes, each node holds one vote. If the two nodes lose connectivity with each other (a "network partition"), neither side can get more than 1 of 2 votes, so neither side can form a quorum.

Without quorum, Proxmox stops all VM operations on both sides. This is the right behavior — it prevents split-brain, where both nodes think they're the primary and start writing conflicting changes. But it means your "HA cluster" just stopped working because of a single network hiccup.

The solution is to add a third vote — something that doesn't run VMs, just exists to break ties. Proxmox calls this a "QDevice."

Solving Quorum with a QDevice on the Synology

I already had a Synology DS920+ NAS in the lab. Synology's Virtual Machine Manager (VMM) lets you run VMs directly on the NAS. So I deployed a tiny Debian VM on the Synology, installed corosync-qnetd, and configured it as the Proxmox cluster's QDevice.

The result: three votes total — pve01, pve02, and the QDevice VM on the Synology. Now if either NUC goes offline, the other two still form a quorum (2 of 3 votes), and the cluster stays operational.

One important implementation detail: I put the QDevice VM on the Management VLAN (192.168.10.x), not the Homelab VLAN (99) where the Proxmox nodes live. That required a cross-VLAN firewall rule to allow the Corosync QDevice protocol through, but it means the QDevice's availability is not coupled to the Homelab VLAN — a partial network failure won't take out both the cluster communication and the tiebreaker.

I chose a Debian VM over using Synology's native package manager (Entware) because Debian gives me standard systemd service management, easy SSH access, and a clean upgrade path. The corosync-qnetd package installs and runs cleanly on Debian without any compatibility gymnastics.

What Went Wrong

The first thing I hit was a cross-VLAN connectivity issue. The QDevice VM was on Management VLAN and the Proxmox nodes were on Homelab VLAN 99. I'd correctly set up the Unifi firewall rules to allow the traffic, but I'd forgotten to account for Corosync using ports 5404 and 5405, not just 5403. The cluster would form but the QDevice wouldn't vote.

journalctl -u corosync-qdevice -n 50 gave me the relevant error. Once I opened the right ports in the Unifi firewall policy, the QDevice connected cleanly.

The second issue was NFS timing. The Synology NFS export that I mount on both Proxmox nodes occasionally races against the Synology boot sequence. On first setup, I'd forget to add a network dependency to the NFS mount service, and the mount would fail silently on reboot. The fix is to add After=network-online.target to the mount unit — straightforward once you know what to look for.

The Failure Modes, Documented

After the cluster was up, I documented the quorum behavior explicitly:

Scenario Cluster State Resolution
pve01 down Quorate (pve02 + QDevice = 2 of 3) Auto-failover
pve02 down Quorate (pve01 + QDevice = 2 of 3) Auto-failover
QDevice down NOT quorate (1 of 3 each side) Restart QDevice VM on Synology
Both NUCs down Offline Manual recovery

The QDevice being a single point of failure (sort of) is a known, accepted risk. If the Synology goes down, Proxmox loses quorum and VMs are fenced — but the Synology also holds the NFS storage, so losing the Synology would already be a bad day. The risk is documented in the risk register.

What This Taught Me

Two-node HA is a specific architecture that requires explicit quorum design. You can't just plug two servers into a switch and call it a cluster — you need to decide in advance what happens when they can't talk to each other, and then build that decision into the system.

The QDevice pattern is the same thing enterprise storage clusters do with "witness" nodes. Learning it in a homelab, where the stakes are low, means the concept is fully internalized before I encounter it in production.
