Ingress, TLS, and Service Discovery

6 May 2026

Tags: homelab, kubernetes, traefik, metallb, cert-manager, tls, dns

Making a Kubernetes service accessible at a real URL with valid TLS involves more moving parts than most tutorials admit. In a cloud environment, some of those parts are handled for you. On bare metal in a homelab, you assemble them yourself. This post traces the full request path from a browser to a pod.

The Full Request Path

When a browser on my laptop loads https://argocd.lab.ls0775.com, here's what actually happens:

  1. DNS resolution: My laptop queries Pi-hole at 192.168.10.10. Pi-hole has a wildcard A record: *.lab.ls0775.com → 192.168.99.100.
  2. TLS handshake: The browser connects to 192.168.99.100:443. That IP is owned by Traefik, which terminates TLS using the lab-wildcard-tls certificate.
  3. Routing: Traefik reads the Host header (argocd.lab.ls0775.com), matches it to an Ingress rule, and proxies the request to the argocd-server Service in the argocd namespace.
  4. Pod: The Service load-balances to one of the Argo CD pods running on a worker node.

Five systems (Pi-hole, MetalLB, Traefik, cert-manager, Kubernetes Ingress) working together to make https://argocd.lab.ls0775.com resolve. Understanding this chain is the difference between debugging "why doesn't the URL work" in 30 seconds versus 30 minutes.

MetalLB: Giving Services a Real IP

As covered in Post 05 — Kubernetes on Bare Metal, MetalLB assigns IPs from a reserved pool to Kubernetes LoadBalancer services. Traefik gets 192.168.99.100 from MetalLB. This is the IP that everything routes through.

The MetalLB configuration defines the pool in Layer 2 mode (ARP-based advertisement, simpler than BGP for a homelab):

apiVersion: metallb.io/v1beta1
kind: IPAddressPool
metadata:
  name: lab-pool
  namespace: metallb-system
spec:
  addresses:
    - 192.168.99.100-192.168.99.149
---
apiVersion: metallb.io/v1beta1
kind: L2Advertisement
metadata:
  name: lab-advertisement
  namespace: metallb-system

Layer 2 mode means MetalLB responds to ARP requests for the pool IPs from the node that currently owns the service. If that node goes down, MetalLB fails over the IP to another node — the failover takes a few seconds, but it's automatic.

Traefik: Ingress and TLS Termination

k3s bundles Traefik as the default ingress controller. Traefik watches the Kubernetes API for Ingress resources and configures routing rules accordingly.

An Ingress resource for Argo CD looks like:

apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: argocd
  namespace: argocd
  annotations:
    traefik.ingress.kubernetes.io/router.entrypoints: websecure
spec:
  tls:
    - hosts:
        - argocd.lab.ls0775.com
      secretName: lab-wildcard-tls
  rules:
    - host: argocd.lab.ls0775.com
      http:
        paths:
          - path: /
            pathType: Prefix
            backend:
              service:
                name: argocd-server
                port:
                  number: 443

Traefik sees this resource, starts listening for requests with the argocd.lab.ls0775.com Host header, and terminates TLS using the lab-wildcard-tls secret. Adding a new service to the cluster requires nothing more than creating an Ingress resource like this one — no changes to Traefik configuration, no changes to DNS.

cert-manager and the Wildcard Certificate

TLS certificates come from cert-manager, a Kubernetes controller that manages the certificate lifecycle. cert-manager integrates with Let's Encrypt to automatically issue and renew certificates.

I'm using a single wildcard certificate for *.lab.ls0775.com. The wildcard covers every service at that domain without requiring a per-service certificate. Let's Encrypt issues wildcard certificates via DNS-01 challenge — proving domain ownership by adding a TXT record to the DNS.

The certificate is stored as a Kubernetes Secret (lab-wildcard-tls) in a central namespace and then copied to application namespaces using a Secret sync mechanism. Every new application namespace that hosts an HTTPS service needs a copy of the secret.

The alternative would be per-service certificates. Let's Encrypt rate limits certificate issuance (50 per domain per week), and managing per-service certificate lifecycle is operationally heavier. One wildcard cert, renewed automatically by cert-manager, is the right choice for a homelab scale.

Pi-hole: Wildcard DNS That Actually Scales

The wildcard DNS record in Pi-hole is what makes service discovery effortless. Pi-hole uses dnsmasq for local DNS, and dnsmasq supports wildcard records:

address=/lab.ls0775.com/192.168.99.100

This single configuration line means every subdomain of lab.ls0775.com resolves to Traefik's MetalLB IP. Adding newapp.lab.ls0775.com requires only an Ingress resource in Kubernetes — no DNS change needed.

The pattern: all resolution goes through Pi-hole, all routing goes through Traefik. Pi-hole doesn't know about individual services. Traefik doesn't care about DNS. Each does one job well.

Deploying hello-world and homepage

The first applications I deployed were deliberately simple: a hello-world app and a homepage dashboard.

The hello-world app is exactly what it sounds like — an nginx container serving a static page, with an Ingress resource pointing hello-world.lab.ls0775.com at it. Its purpose was to validate the full stack end-to-end before deploying anything more complex.

The homepage app is a gethomepage.dev dashboard — a bookmark page with service status widgets. It's served at home.lab.ls0775.com and provides a single-page view of the cluster's running services. Not a critical workload, but useful as a visual index of what's deployed.

Both apps have READMEs in k8s/apps/ describing their configuration. Writing a per-app README is a discipline I'm enforcing on myself — it forces me to document what the app is, what it depends on, and how to access it.

What Went Wrong

The trickiest issue I hit was the Argo CD grpc routing. Argo CD's CLI (argocd login) uses gRPC over HTTP/2. Traefik 2.x has some configuration requirements for gRPC passthrough that aren't in the default setup — specifically, the backend needs to be configured with the h2c protocol. Without this, the CLI works but hangs on some operations. The fix was an annotation on the Ingress resource (traefik.ingress.kubernetes.io/service.servertransport). It took a while to find because the browser UI worked fine — only the CLI was affected.

What This Taught Me

The wildcard pattern — wildcard DNS + wildcard TLS + Traefik routing on Host header — is a pattern I've seen in production environments and never fully understood until I built it myself. Now it makes complete sense: you separate the "where does traffic land" problem (DNS + load balancer) from the "where does traffic go" problem (ingress routing), and each layer operates independently.
