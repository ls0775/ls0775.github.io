Network Segmentation: 6 VLANs for a Home Network

15 April 2026

Tags: homelab, networking, vlans, unifi, pihole, dns

Most homelab guides treat networking as an afterthought — "plug everything into your router, move on." I wanted to do it properly from day one, which meant VLANs, firewall policies, and a DNS strategy that would actually scale as I added services. This post covers that setup and the specific decision that changed how I think about home network security.

Why Segment at All?

Before I started, all my home devices were on one flat network. My personal laptop, my security cameras, my IoT devices, my lab servers — all on the same subnet, all able to talk to each other freely. That's the default for most home routers.

There are two good reasons to change this:

Security. IoT devices have a bad track record. A compromised smart bulb or camera shouldn't have free access to your NAS or your development machines. VLANs let you enforce network-layer isolation between device categories.

Lab isolation. When you're experimenting with infrastructure — rebooting servers, reconfiguring networking, deploying things that might misbehave — you don't want that noise affecting your personal devices. Running the lab on its own VLAN means a misconfigured firewall rule or a crashed Kubernetes node doesn't take down your home network.

The Six VLANs

I ended up with six VLANs, managed by a UniFi Cloud Gateway Fiber (UCG Fiber) as the router:

VLAN Name Subnet What's Here
1 Management 192.168.10.0/24 Pi-hole, Synology NAS, QDevice VM
20 Trusted 192.168.20.0/24 Personal PCs, phones
30 Guest 192.168.30.0/24 Guests, Chromecasts
40 IoT 192.168.40.0/24 Home Assistant, smart devices
50 CCTV 192.168.50.0/24 Security cameras
99 Homelab 192.168.99.0/24 Proxmox nodes, k3s cluster, lab services

VLAN 99 for the Homelab was a deliberate choice — a high, memorable number that stands out in firewall logs and makes it obvious which VLAN I'm looking at. The Management VLAN being VLAN 1 follows standard convention: it's the default untagged VLAN on most switches, which makes physical cable management simpler.

The Move That Changed How I Think About Home Networks

When I initially set up the lab, I put everything — Proxmox, the Synology, the NUCs — on VLAN 20 (Trusted). It was the path of least resistance.

Then I sat down to write the firewall policy and realized the problem: if I put lab servers on the Trusted VLAN, the security boundary between "lab experiments" and "personal data" is zero. My Kubernetes cluster was one misconfigured pod security policy away from being on the same network as my financial documents.

So I migrated all lab infrastructure to VLAN 99 (completed April 9, 2026). The Proxmox nodes moved to 192.168.99.11/12. The k3s VMs moved to 192.168.99.21/22/30/31. The Synology stayed on Management VLAN because it serves both the lab (NFS storage) and personal use (backups), and putting it on both VLANs with selective firewall rules was cleaner than duplicating it.

The migration was straightforward in Unifi — reassign the switch port VLAN tags and update the static IP reservations. The more interesting challenge was getting the firewall rules right afterward.

The IP Allocation Strategy

For VLAN 99, I designed the IP range semantically rather than just assigning IPs as devices appeared:

Range Purpose
.1 Gateway
.11–.19 Proxmox bare-metal hosts
.20–.29 k3s control-plane nodes
.30–.39 k3s worker nodes
.40–.99 Other infra VMs (Gitea, CI runners, etc.)
.100–.149 MetalLB pool (Kubernetes LoadBalancer IPs)
.200–.249 DHCP dynamic pool

This design means the IP address tells you what the device is. 192.168.99.22 is obviously a k3s server node. 192.168.99.100 is obviously a MetalLB-assigned service. Firewall rules written against IP ranges are self-documenting in a way that per-host rules are not.

Firewall Policy: Default Deny

All inter-VLAN traffic is blocked by default in Unifi. Permitted flows are explicitly enumerated:

From To Port Why
All VLANs Pi-hole (192.168.10.10) 53/UDP,TCP DNS for every VLAN
Homelab (99) Synology (192.168.10.11) 2049/TCP NFS storage backups
Homelab (99) QDevice VM Corosync ports Proxmox cluster quorum

The principle: everything is blocked unless there's a specific, documented reason to allow it. This is the same "default deny" posture used in enterprise network security. In practice for a homelab, it means I occasionally have to debug a connectivity issue when I add a new service, but that debugging always reveals a gap in my mental model that I'm glad to know about.

Pi-hole as the Network DNS

Pi-hole sits at 192.168.10.10 and is configured as the primary DNS server for all VLANs. Every device on the network — across all six VLANs — uses Pi-hole as its resolver.

Pi-hole gives me three things:

  1. Network-wide ad blocking — effective and centralized, no per-device configuration
  2. Local DNS records — I can define pve01.lab.ls0775.com → 192.168.99.11 and have that resolve from any device on any VLAN
  3. Query logging — invaluable for debugging connectivity issues ("is DNS resolving? is the response correct?")

For lab services, I adopted a wildcard DNS pattern: *.lab.ls0775.com resolves to 192.168.99.100 (the Traefik MetalLB IP). This means I can add a new Kubernetes service with an Ingress rule and it gets a hostname automatically — no Pi-hole configuration change needed. The first time I deployed a new app and it just appeared at newapp.lab.ls0775.com, that felt genuinely satisfying.

What Went Wrong

The first issue was forgetting the NFS firewall rule when I moved the lab to VLAN 99. After the move, the Proxmox nodes couldn't reach the Synology NFS export. The error in the Proxmox storage UI was unhelpfully generic ("storage unavailable"), but ping 192.168.10.11 from pve01 confirmed the problem was reachability, not the NFS configuration itself. Added the firewall rule, NFS came back.

The second issue was DNS forwarding. When I set up Pi-hole on VLAN 1 (Management) and pointed all VLANs at it, I initially forgot to add a Unifi firewall rule allowing DNS traffic from VLAN 30 (Guest) to VLAN 1. Guests couldn't resolve DNS. The fix was a 30-second firewall rule change, but debugging it took longer because I was looking at the wrong VLAN.

What This Taught Me

Default-deny firewall policy forces you to think explicitly about every communication path in your network. That's uncomfortable when you first set it up — it feels like everything breaks and nothing works. But it teaches you your network in a way that default-allow never will. Three months later, I can draw the inter-VLAN firewall rules from memory because I had to build them one at a time and understand why each one exists.
