Ansible Inventory Design and Finding Configuration Drift

19 May 2026

Tags: homelab, ansible, configuration-management, idempotency, devops, drift-detection, k3s

I thought Ansible was keeping my servers consistent. Then I ran a diff of manually installed packages between two supposedly identical control-plane nodes and found one had helm and the other didn't. Nobody had noticed. This post is about how that happened, how to systematically find it, and how to prevent it from happening again.

How Drift Happens

Drift is what happens when the actual state of your infrastructure diverges from the state described in your code. In this homelab, the pattern is familiar: you SSH into a server to fix something urgently, install a package to test a theory, and then move on. The fix works. You forget to update the playbook.

Six months later, you provision a second server from the playbook. It doesn't have the package. Things work differently between the two nodes, and the reason isn't obvious because the two playbooks are identical — the drift lives outside both of them.

Helm was the example here. I had installed it manually on k3s-server-01 early on, long before the Ansible playbook for that server was written. The playbook was written later, based on what I thought the server needed — but helm never made it in. k3s-server-02 was provisioned entirely from the playbook and came out without helm. Two supposedly equivalent control-plane nodes, subtly different.

Finding Manual Drift

There are two dimensions of drift to find: things Ansible knows about (has tasks for) but where the live state doesn't match, and things Ansible doesn't know about at all.

What Ansible doesn't know about

apt-mark showmanual lists packages that were explicitly installed by a human (not pulled in as a dependency). Diffing two servers immediately shows what's inconsistent:

diff <(ssh k3s-server-01 "apt-mark showmanual | sort") \
     <(ssh k3s-server-02 "apt-mark showmanual | sort")

The output in this case showed that k3s-server-01 had helm, btop, gh, gitleaks, gnupg, build-essential, pipx, and tofu — none of which were in any playbook. Some of those (like btop) belonged on the control node workstation, not the server. Others (like helm) were genuinely missing from the server playbooks.

The same check on apt sources reveals repos that were added manually:

diff <(ssh k3s-server-01 "ls /etc/apt/sources.list.d/ | sort") \
     <(ssh k3s-server-02 "ls /etc/apt/sources.list.d/ | sort")

What Ansible knows about but hasn't applied

ansible-playbook --check runs a playbook in dry-run mode — it reports what would change without making any changes. Every task showing changed is real drift:

ansible-playbook --check -i inventory/hosts.yaml playbooks/k3s-server-01.yaml

Running this revealed six items I hadn't known about:

  • The etcd snapshots directory had 0700 permissions; the playbook wanted 0750
  • The k3s token file had 0600; the playbook wanted 0640 (group-readable by k3s-admins)
  • The etcd rsync cron was set to fire at minute 0; the playbook specified minute 30
  • The cron itself had no Ansible name tag — it was added manually, outside Ansible's awareness
  • A sudoers rule (/etc/sudoers.d/ansible) existed on k3s-server-01 but not k3s-server-02
  • /etc/rancher/k3s/config.yaml would have been overwritten on re-run (a playbook bug — see below)

Fixing the Playbook Bugs

Two bugs in the server playbooks were found during this process.

Bug 1: The config.yaml overwrite problem. The server bootstrap playbook used ansible.builtin.copy with hardcoded content for /etc/rancher/k3s/config.yaml. A separate k3s-enable-metrics.yaml playbook (run later) adds a metrics block to the same file using blockinfile. If the bootstrap playbook was ever re-run, it would silently overwrite the metrics block — and copy has no concept of "leave existing content alone."

The fix: force: false on the copy task. This tells Ansible to only write the file if it doesn't already exist. The bootstrap playbook creates the initial config; the metrics playbook manages its own block after that.

- name: Write k3s config.yaml
  ansible.builtin.copy:
    dest: /etc/rancher/k3s/config.yaml
    force: false
    content: |
      ...

Bug 2: Orphaned cron entries. The rsync cron for etcd snapshots was added manually with no Ansible name tag. Ansible's cron module identifies entries by a name comment (#Ansible: <name>). Without it, Ansible doesn't know the cron exists and would add a second, duplicate entry on the next run. The fix: delete the orphaned entry on each server, then let Ansible add the named version.

Evolving the Inventory Structure

The original inventory was flat — all k3s servers in one k3s_servers group, all agents in k3s_agents. This caused a problem: k3s-server-03 is a quorum device running on a Synology VM, not a full control-plane node. Running the server playbook (which installs helm, configures NFS, sets up etcd snapshot rsync) against it would be wrong.

Ansible supports hierarchical groups natively:

k3s_servers:
  children:
    k3s_control_plane:
      hosts:
        k3s-server-01:
          ansible_host: 192.168.99.21
        k3s-server-02:
          ansible_host: 192.168.99.22
    k3s_qdevice:
      hosts:
        k3s-server-03:
          ansible_host: 192.168.10.23

With this structure, k3s_servers still matches all server nodes (useful for ansible k3s_servers -m ping), but playbooks can now target k3s_control_plane specifically. The same pattern applies to Proxmox — proxmox_nodes for the full Proxmox VE hosts, proxmox_qdevice for the Ubuntu VM running the quorum daemon.

The final inventory structure:

all
├── control_node         (localhost — workstation)
├── k3s_servers
│   ├── k3s_control_plane   (k3s-server-01, 02)
│   └── k3s_qdevice         (k3s-server-03)
├── k3s_agents           (k3s-agent-01, 02)
└── proxmox
    ├── proxmox_nodes    (pve01, pve02)
    └── proxmox_qdevice  (pve03)

The k3s-enable-metrics.yaml playbook, which had been targeting k3s_servers, was updated to target k3s_control_plane — metrics apply only to the full control-plane nodes, not the quorum device.

Codifying What Was Missing

After finding the drift, the fixes went into the playbooks:

Helm was added to both server playbooks with the full apt repo setup — key download, source file, package install. Idempotent because the key task uses creates: (only runs if the keyring doesn't exist) and the apt task is always idempotent by default.

Sudoers rule was added as a task using ansible.builtin.copy with validate: /usr/sbin/visudo -cf %s. The validate parameter runs visudo on the file before writing it — if the syntax is invalid, Ansible aborts without writing a broken sudoers file. A scoped rule (not a broad NOPASSWD:ALL) lets the ansible user read the k3s token without full root access:

- name: Add scoped sudoers rule for Ansible token access
  ansible.builtin.copy:
    dest: /etc/sudoers.d/ansible
    content: "ls0775 ALL=(root) NOPASSWD: /usr/bin/cat /var/lib/rancher/k3s/server/token\n"
    mode: '0440'
    validate: /usr/sbin/visudo -cf %s

The broad NOPASSWD:ALL rule — which makes become: true work without --ask-become-pass on the servers — comes from cloud-init, not Ansible. It's set in the OpenTofu VM provisioning config and applied at first boot. That's the right place for it: it's a provisioning concern, not a configuration management concern.

The ansible.cfg Default

One small quality-of-life improvement: an ansible.cfg file in the config/ directory:

[defaults]
inventory = inventory/hosts.yaml
deprecation_warnings = False
host_key_checking = False

[privilege_escalation]
become = True
become_method = sudo

inventory sets the default so -i inventory/hosts.yaml no longer needs to be typed on every command. host_key_checking = False prevents new hosts from blocking on the SSH fingerprint prompt — useful when adding new nodes to inventory.

The Workflow Going Forward

The discipline is: after any manual change on a server, update the playbook before closing the session. The check mode acts as a verification:

# Did I capture everything?
ansible-playbook --check playbooks/k3s-server-01.yaml
ansible-playbook --check playbooks/k3s-server-02.yaml
# Expected: changed=0 on both

changed=0 means the playbook fully describes the live state. Any non-zero count is a prompt to investigate — either the playbook is missing something, or the live server has drift that needs to be either codified or reverted.

What This Taught Me

Drift is invisible until you look for it. The two servers had been running in production (or homelab production) for weeks with this inconsistency, and nothing had broken — because the workloads running on them didn't need helm on the node. But the next time a server was rebuilt from the playbook, it would be subtly different from the existing one in a way that would have been hard to diagnose.

The check mode workflow is the equivalent of a linter for infrastructure state. Running it routinely is cheap. Discovering drift in production during an incident is expensive.
