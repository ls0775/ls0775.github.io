Kubernetes on Bare Metal: Choosing and Deploying k3s

22 April 2026

Tags: homelab, kubernetes, k3s, bare-metal, metallb, traefik

Running Kubernetes in a homelab is different from running it in a cloud. There's no managed control plane, no load balancer service backed by an AWS ALB, no persistent volume provisioner that just works. You have to solve those problems yourself — and solving them is exactly where the learning happens.

Why k3s Over kubeadm

The standard way to stand up a self-managed Kubernetes cluster is kubeadm. It's the official Kubernetes tool for cluster initialization and is widely documented. I chose k3s instead.

k3s is a lightweight, CNCF-certified Kubernetes distribution maintained by Rancher (now SUSE). It packages the Kubernetes control plane into a single binary, includes Traefik as the default ingress controller and Flannel as the default CNI, and handles certificate management automatically. That's a lot of operational overhead that kubeadm makes you solve yourself.

For a production environment, I'd think harder about this choice — there are trade-offs in the k3s defaults that you might want to customize. But for a homelab where the goal is to learn GitOps, observability, and platform operations (not to learn how to wrangle kubeadm bootstrap tokens), k3s gets you to a working cluster faster and keeps it working with less daily maintenance.

The other consideration: k3s is what I'd reach for in a resource-constrained environment at work (edge deployments, small clusters, iot). Learning its operational characteristics isn't just homelab knowledge — it's career knowledge.

Why VMs Over LXC

Proxmox supports both full VMs (KVM) and containers (LXC). LXC containers are lighter — faster to start, lower overhead. It was tempting to run k3s inside LXC containers instead of full VMs.

I didn't. Here's why:

k3s runs container workloads. A container runtime inside a container (LXC) requires privileged containers or specific kernel capability grants. It works, but it's a known footgun — you end up fighting the container isolation model to give k3s what it needs. More importantly, running Kubernetes nodes in LXC is architecturally different from what you'd do in production, which always uses full VMs or bare metal. If the goal is to learn production-like infrastructure, using a shortcut that doesn't exist in production undermines the point.

Full VMs also give me proper kernel isolation — each k3s node gets its own kernel, which matters when you're testing things that interact with kernel-level features like eBPF, cgroups v2, or kernel network namespaces.

The Cluster Topology

Four VMs, spread across the two Proxmox hosts:

Node Role Host IP VM ID
k3s-server-01 control-plane + etcd pve01 192.168.99.21 301
k3s-server-02 control-plane + etcd pve02 192.168.99.22 304
k3s-agent-01 worker pve01 192.168.99.30 302
k3s-agent-02 worker pve02 192.168.99.31 305

Each VM: 4 vCPU, 8GB RAM, 80GB disk. The server nodes run the control plane (API server, etcd, scheduler, controller-manager). The agent nodes run workloads.

Splitting the VMs across hosts means that losing pve01 takes down k3s-server-01 and k3s-agent-01, but k3s-server-02 stays up on pve02 and the control plane keeps running. This is the same "spread across failure domains" principle used in cloud Kubernetes deployments — except instead of availability zones, my failure domains are two NUCs on a shelf.

MetalLB: The Problem Kubernetes Has on Bare Metal

In a cloud environment, when you create a Kubernetes Service of type LoadBalancer, the cloud provider automatically provisions a load balancer (AWS ALB, GCP CLB, etc.) and assigns it an external IP. On bare metal, nothing does that for you. By default, a LoadBalancer service just sits in <pending> forever.

MetalLB solves this. It's a bare-metal load balancer implementation that watches for LoadBalancer services and assigns them IPs from a pool you define. I configured it with the IP range 192.168.99.100–192.168.99.149 (reserved in the VLAN 99 IP allocation from Post 03 — Network Segmentation).

Traefik — k3s's bundled ingress controller — gets IP 192.168.99.100 from MetalLB. Pi-hole's wildcard DNS record (*.lab.ls0775.com) points to that IP. The result: any Kubernetes service with an Ingress rule gets a publicly-resolvable (within the lab) hostname automatically, with no additional infrastructure changes.

The first time I deployed an app and it appeared at appname.lab.ls0775.com with working TLS, that was a genuinely satisfying moment.

Initial Deployment

The initial k3s installation on k3s-server-01 was a single command:

curl -sfL https://get.k3s.io | sh -s - server \
  --cluster-init \
  --tls-san k3s-server-01.lab.ls0775.com \
  --tls-san 192.168.99.21

--cluster-init tells k3s to use etcd (not SQLite) and prepare for additional server nodes. --tls-san adds subject alternative names to the API server's TLS certificate so it's valid when accessed via hostname or IP.

After installation, the kubeconfig at /etc/rancher/k3s/k3s.yaml has the cluster credentials. I copied that to my workstation and adjusted the server address to point at the cluster hostname instead of localhost.

kubectl get nodes
# k3s-server-01   Ready   control-plane,master

One node, working. Not HA yet, but a good starting point to verify the network plumbing was right before complicating it.

What Went Wrong

The main issue I hit early was TLS certificate validation. When I tried to connect to the cluster from my workstation, kubectl rejected the certificate because the server address in the kubeconfig (192.168.99.21) wasn't in the TLS certificate's SAN list — I'd initially deployed without the --tls-san flag. Fix: redeploy k3s with the correct SANs, regenerate the kubeconfig. Not catastrophic, just a restart of the k3s service.

The second issue was MetalLB configuration. MetalLB requires a ConfigMap (or CRD, depending on version) that defines the IP pool. I'd deployed MetalLB but not configured the pool, so LoadBalancer services still showed <pending>. Reading the MetalLB docs more carefully fixed it — the error messages weren't helpful, but the documentation was.

What This Taught Me

Bare-metal Kubernetes forces you to understand the layers that cloud providers abstract away. A LoadBalancer service in EKS "just works" because AWS is running MetalLB-equivalent infrastructure for you behind the scenes. Running MetalLB yourself makes the underlying mechanism explicit — and once you understand it, cloud load balancers are much less magical.

The same applies to ingress: Traefik is doing exactly what AWS ALB Ingress Controller does, just without the AWS-specific annotations.
