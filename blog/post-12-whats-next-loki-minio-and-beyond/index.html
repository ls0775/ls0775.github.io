What's Next: Loki, MinIO, and the Road to Phase 5

16 May 2026

Tags: homelab, loki, minio, kubernetes, sre, devops, reflection

This is the last post in the "building the platform" arc of this series. The cluster is operational, GitOps is wired up, monitoring is running. But operational doesn't mean done — there are two Critical-severity gaps that need addressing, and a Phase 5 plan for closing the remaining architectural debt. This post covers what's left and what I've learned from the build so far.

The Two Critical Gaps

R-002: OpenTofu State on a Single Workstation

The infra/terraform.tfstate file lives on my DevOps workstation. This file is OpenTofu's record of everything it has created — the mapping between HCL declarations and actual Proxmox VMs. Without it, OpenTofu can't manage the infrastructure it created. If my workstation dies (drive failure, theft, a really bad rm -rf mistake), I lose the ability to make OpenTofu changes cleanly. Recovery is possible via tofu import, but it's tedious and error-prone for a multi-VM setup.

The fix is a remote state backend — a central server that stores the state file, accessible from any machine, with locking to prevent concurrent applies from corrupting it.

The tool for this in a self-hosted environment is MinIO — an open-source object storage server that exposes an S3-compatible API. OpenTofu's S3 backend works with MinIO without modification:

terraform {
  backend "s3" {
    bucket = "tofu-state"
    key    = "homelab/terraform.tfstate"
    region = "us-east-1"  # required field, arbitrary for MinIO
    
    endpoint                    = "https://minio.lab.ls0775.com"
    skip_credentials_validation = true
    skip_metadata_api_check     = true
    force_path_style            = true
  }
}

MinIO will be deployed as a Kubernetes workload in the storage namespace, using the Synology NFS mount for persistence. Once it's up, migrating the state backend is a one-time tofu init -reconfigure that copies the existing state file to the new backend.

This is Priority 1 for Phase 4.

R-004: No Log Aggregation

Prometheus tells you what is happening right now. It doesn't tell you what was happening when the pod crashed at 3am last Tuesday. Without log aggregation, historical troubleshooting is impossible — you're limited to whatever the running container currently has in its log buffer.

Loki is the log aggregation system built to complement Prometheus. It stores log streams indexed by labels (namespace, pod name, container) and exposes a query language (LogQL) that mirrors PromQL syntax. Grafana has native Loki support — you can correlate a spike in error metrics with the log lines from the same time window in a single dashboard.

Promtail is the log shipping agent — a DaemonSet that runs on every node, reads container logs from /var/log/pods/, and ships them to Loki.

The deployment plan:

  1. Deploy Loki in the monitoring namespace (same as Prometheus/Grafana)
  2. Deploy Promtail as a DaemonSet on all nodes
  3. Add Loki as a Grafana datasource
  4. Write initial LogQL queries for the most useful views: error rates by namespace, k3s API server logs, etcd logs

Once Loki is deployed, the observability stack is metrics + logs. The third pillar (distributed tracing) is out of scope for this project's scale — tracing is most valuable in microservices architectures with complex call graphs, which doesn't describe a homelab with a handful of apps.

Phase 5: Closing the Remaining Debt

R-001: Third etcd Node

The 2-node etcd cluster is the biggest architectural debt. It's documented, understood, and accepted as a temporary state — but it's temporary.

The plan: deploy a third k3s server node as a VM on the Synology VMM. This node will be:

  • Tainted — marked so no application workloads schedule on it (only control-plane components)
  • Lightly resourced — enough for etcd replication (2 vCPU, 2GB RAM), not sized for workloads
  • Cross-failure-domain — running on the Synology rather than pve01 or pve02, so losing either NUC still leaves two etcd members

With three etcd members, the cluster can tolerate losing any single member (including the Synology VM) and continue operating. That's true HA for the control plane.

R-003: Off-Site Backups

The Synology NAS holds etcd snapshots, VM backups, and the only copy of several years of personal data. A house-level event (fire, power surge, flood) would wipe all of it simultaneously.

The fix: encrypted off-site sync. The candidates are AWS S3 (Glacier or Intelligent-Tiering for cost optimization), Backblaze B2 (cheaper per-GB than S3), or Cloudflare R2 (free egress). The implementation is a scheduled backup job that encrypts with age or GPG before uploading, so the cloud provider never has plaintext data.

R-006: GitHub Actions for Infrastructure CI/CD

Currently, tofu plan and tofu apply run manually from my workstation. There's no automated check that pull requests are consistent with the Proxmox state, and no guardrails against applying a bad change without review.

The Phase 5 plan: GitHub Actions pipeline that runs tofu plan on every PR targeting main and posts the plan output as a PR comment. tofu apply remains a manual step (requiring deliberate approval) but the plan check is automated. This brings the infrastructure workflow in line with how most organizations handle IaC — automated plan, human-approved apply.

What I'd Do Differently

If I were starting this project again from scratch, three things would change:

MinIO first. The remote state backend should be deployed before the first tofu apply, not as a retrofit. Migrating state backends is straightforward, but starting with local state creates an unnecessary window of risk.

Three etcd nodes from the start. The 2-to-3 node etcd migration is not difficult, but it would have been easier to design for 3 nodes upfront than to add the third one to an existing 2-node cluster. The Synology VMM was always available; using it from day one for the third control-plane node would have eliminated R-001 entirely.

Loki alongside Prometheus. I deployed Prometheus first and planned to add Loki "later." Later has been several weeks. Deploying them together from the start would have avoided several incidents where I had no historical log context for troubleshooting.

What This Project Taught Me

The technology is the easy part. Kubernetes, Prometheus, Ansible, Argo CD — these are well-documented tools with active communities. The hard part is the operational discipline: deciding what the right architecture is, documenting why, tracking what's broken, and building the habits that make a system manageable over time.

The single most valuable habit from this project is writing things down immediately — ADRs when you make a decision, risk register entries when you identify a gap, handoff log entries when you end a session. The compounding effect of that documentation is that six months from now, I can come back to this system and understand it. Without the documentation, I'd be reverse-engineering my own infrastructure.

This series documents a real build, with real gaps. R-001, R-002, R-003, R-004 — these are the things I'm not proud of, the places where the architecture isn't where I want it. Putting them in a public blog post rather than hiding them is deliberate: a portfolio piece that only shows the polished version is less credible than one that demonstrates you know what's wrong and have a plan to fix it.
