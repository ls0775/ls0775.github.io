Post-Provisioning Configuration with Ansible

29 April 2026

Tags: homelab, ansible, configuration-management, k3s, devops, idempotency

OpenTofu creates the VM. It does not configure what runs inside it. That's a separate concern — and Ansible is how I handle it. This post is about the clean separation between infrastructure provisioning (IaC) and configuration management, and why that separation matters.

The OpenTofu + Ansible Split

There's a natural boundary in infrastructure management:

OpenTofu manages infrastructure state: "There should be a VM on pve01 with 4 vCPUs, 8GB RAM, and IP 192.168.99.21. That VM should be running."

Ansible manages software state: "That VM should be running k3s in server mode, joined to the existing cluster, with the correct token."

These are different concerns. OpenTofu speaks to the Proxmox API — it doesn't know or care what's running inside the VM. Ansible speaks to the VM over SSH — it doesn't know or care that the VM was created by OpenTofu rather than a GUI wizard.

Keeping them separate means each tool does what it's best at. OpenTofu's declarative model is excellent for resources with clean create/update/delete semantics. Ansible's task-based model is excellent for configuration that's stateful in ways that don't map cleanly to a single API call.

Why Ansible

For post-provisioning configuration, I looked at a few options:

Cloud-init handles initial boot configuration (hostname, SSH keys, initial package installs). I use cloud-init for the basics, but it only runs once (on first boot) and isn't idempotent in any meaningful sense.

Shell scripts work, but they're fragile. A script that installs k3s and joins a node is easy to write and hard to re-run safely. Did the join already happen? Is the token still valid? You end up adding a lot of "if already done, skip" logic that Ansible handles for you.

Ansible is idempotent by design. Every Ansible task is supposed to describe a desired state, and running the playbook twice should be safe — the second run detects that the state is already correct and makes no changes. For joining k3s nodes (where joining an already-joined node is an error), that idempotency requires some care, but the pattern is well-established.

I also chose Ansible over Puppet or Chef because it's agentless — no daemon to install or manage on the target hosts. The control machine (my workstation) SSH-es into the target and runs tasks. For a homelab where nodes get rebuilt occasionally, not having an agent dependency is significantly simpler.

The Inventory Structure

Ansible knows about your infrastructure through an inventory file. Mine is YAML and lives at config/inventory/hosts.yaml:

all:
  children:
    k3s_servers:
      hosts:
        k3s-server-01:
          ansible_host: 192.168.99.21
        k3s-server-02:
          ansible_host: 192.168.99.22
    k3s_agents:
      hosts:
        k3s-agent-01:
          ansible_host: 192.168.99.30
        k3s-agent-02:
          ansible_host: 192.168.99.31

Groups (k3s_servers, k3s_agents) let me target playbooks at the right set of hosts. When I want to join a new agent node, I run the agent playbook and Ansible knows exactly which hosts are agents.

The k3s Node-Joining Playbooks

The most important playbooks are for joining k3s nodes to the cluster. The pattern for joining a server node (k3s-server-02):

  1. Fetch the cluster join token from k3s-server-01 via SSH
  2. Install k3s in server mode on k3s-server-02, passing the token and the existing server's address
  3. Verify the node appears in kubectl get nodes with the correct role

The token retrieval step is important: rather than having the token hardcoded in the playbook (bad — it's a secret) or requiring the operator to look it up manually (tedious and error-prone), the playbook reads it directly from the source:

- name: Get k3s join token
  shell: cat /var/lib/rancher/k3s/server/node-token
  register: k3s_token
  delegate_to: k3s-server-01

- name: Join k3s server node
  shell: |
    curl -sfL https://get.k3s.io | sh -s - server \
      --server https://192.168.99.21:6443 \
      --token {{ k3s_token.stdout }} \
      --tls-san {{ inventory_hostname }}.lab.ls0775.com
  when: "'control-plane' not in existing_roles.stdout"

The when condition on the join step is the idempotency guard — if the node is already a control-plane member, skip the join. That check queries kubectl get nodes first and registers the result.

Running the Playbooks

cd config/

# Ping all hosts to verify SSH connectivity
ansible all -i inventory/hosts.yaml -m ping

# Join all agent nodes
ansible-playbook -i inventory/hosts.yaml playbooks/k3s-agent.yaml

# Join a specific agent node
ansible-playbook -i inventory/hosts.yaml playbooks/k3s-agent.yaml --limit k3s-agent-02

The --limit flag is useful when adding a new node — you've already added it to the inventory, and you want to run the playbook against just the new node without touching the existing ones.

What Went Wrong

The main issue I hit was SSH key authentication. Ansible connects via SSH, and the first time I ran a playbook against a new VM, it failed because the SSH key wasn't authorized on the target. The fix is to add the SSH public key to the VM's authorized_keys during provisioning — which is actually a cloud-init job, not an Ansible job. Once I set up cloud-init templates in Proxmox to include the SSH key, new VMs were immediately reachable by Ansible without any manual setup.

The second issue was Ansible's Python dependency. Ansible tasks run Python on the target host, and some of my early Ubuntu VMs didn't have Python installed. The fix: add apt-get install -y python3 to the cloud-init user-data. It's a chicken-and-egg problem (you need Ansible to run to install Python, but you need Python to run Ansible), solved by doing the Python install at cloud-init time instead.

What This Taught Me

The IaC + Ansible split clarifies something that gets blurry when you use tools like Terraform with remote-exec provisioners or Ansible with the Terraform provider: infrastructure state and configuration state are different things with different lifecycle properties.

Infrastructure changes slowly — a VM doesn't get recreated every time you change an app config. Configuration changes more frequently — you might update the k3s version, add a node, or change a configuration parameter without touching the VM's existence or resource allocation. Keeping these separate means changes in one domain don't accidentally trigger unwanted changes in the other.
