Running a Homelab Like a Production Platform

13 May 2026

Tags: homelab, sre, documentation, adr, risk-register, devops, operations

The technology in this homelab — Proxmox, k3s, Argo CD, Prometheus — is the easy part to talk about. Every major component has tutorials and documentation. What I want to write about this week is the operational discipline layer: the practices borrowed from enterprise SRE that turn a collection of services into a platform. These are the things that differentiate "I have a homelab" from "I operate a homelab."

Architecture Decision Records

There are 15 ADRs in this repository. An ADR is a short document that records a non-obvious technical decision: what was decided, why, and what trade-offs were accepted.

Some examples from this project:

  • ADR-001: Proxmox VE over VMware ESXi — open source, API-driven, no licensing concerns post-Broadcom acquisition
  • ADR-004: k3s over kubeadm — lower operational overhead, faster path to learning GitOps and SRE
  • ADR-008: OpenTofu over Terraform — open-source fork, Linux Foundation governance, full compatibility
  • ADR-012: k3s SQLite → etcd — SQLite is a SPOF for the control plane; etcd enables multi-server HA
  • ADR-014: Single Source of Truth documentation model — stops IP addresses and status flags from duplicating across 10 files

The value of ADRs is not in the writing — it's in the future reading. Six months from now, when I'm looking at a configuration and wondering "why did I choose this?" the ADR answers that. When an AI coding assistant picks up the repository, the ADRs provide the "why" context that code alone can't.

Most importantly: ADRs are written immediately after the decision, while the context is fresh. Trying to reconstruct the reasoning six months later produces a post-hoc rationalization, not an accurate record. The discipline is writing the ADR before moving on, even when the task feels done.

The Risk Register

The risk register is a document I didn't expect to use seriously in a homelab — it felt like an enterprise formality. I was wrong.

The homelab has seven documented risks:

ID Risk Severity
R-001 2-node etcd quorum is not truly HA Critical
R-002 OpenTofu state file lives on one workstation Critical
R-003 All backups are on-site (Synology) High
R-004 No log aggregation (Loki not deployed) High
R-005 SLOs defined but not automatically measured Medium
R-006 No CI/CD for infrastructure (manual tofu apply) Medium
R-007 Static x509 certs (no centralized identity) Low

Writing these down did several things. First, it forced me to be explicit about what I know is broken. R-001 and R-002 are Critical risks that I'm operating with knowingly — that's different from not knowing about them. Second, it gave me a prioritized backlog: R-002 (MinIO for remote state) and R-004 (Loki) are the highest-priority unfinished work. Third, it gave me something to show an employer that demonstrates I think about reliability beyond "does it work right now?"

The risk register lives in docs/operations/risk-register.md and is updated when risks are mitigated or new risks are identified. It's not a document you write once and forget — it's an active artifact.

Single Source of Truth Documentation

Documentation duplication is a bug. The same IP address appearing in five different files is the documentation equivalent of copy-pasted code — the moment one copy gets updated, the others drift out of sync.

ADR-014 established a Single Source of Truth model for this repo:

  • docs/environment-data.md — canonical location for all IPs, VLAN IDs, hardware specs, hostnames. Every other document links here rather than repeating the values.
  • PLAN.md — canonical location for project status, phase tracking, what's done, what's next.
  • docs/adr/ — canonical location for architectural decisions.
  • docs/archive/phase-*/ — historical build logs (how things were set up), not kept current.

This feels obvious when you say it, but implementing it required moving content around and adding the discipline of "don't put facts here, link to the SSoT." The payoff is real: when I added a k3s worker node, I updated docs/environment-data.md once, and every other document that links to it is automatically current.

The Handoff Log

This is one of the practices I'm most proud of, and the one that most directly demonstrates a production SRE mindset.

In professional SRE, "on-call handoff" is a formal process. The engineer ending their on-call shift documents what happened, what's still in progress, and what the next engineer needs to know to operate safely. This prevents the next person (or the next AI agent session) from wasting time reconstructing context that the previous shift already had.

ADR-015 formalized a daily platform handoff log for this homelab at docs/operations/handoff-log.md. Each entry covers:

  1. Significant changes made
  2. Unfinished business and next steps
  3. Active blocking risks
  4. Platform status (cluster up? backups firing? any observability gaps?)

The primary audience is my future self (or the AI agents I work with) picking up the project after time away. The format treats that future self like a new team member who needs context to operate safely, not like someone who remembers everything from last time.

The discipline: write the handoff entry before closing the session, even if it's short, even if nothing major happened. The habit of writing context down is more valuable than any individual entry.

AI Agents as Engineering Collaborators

This project is built with AI coding assistants as active collaborators — primarily Claude Code, with some Gemini sessions. I want to be transparent about this rather than presenting the work as purely solo.

The AI does not design the architecture or make decisions. It implements, documents, and reasons about the implications of choices I've already made. The Architecture Decision Records, risk register, and SLO definitions are all decisions I made; the AI helped articulate them and implement the documentation structure.

What AI assistance changes is the operational pattern: I can pick up the project after a week off, give the AI the handoff log and relevant ADRs as context, and continue efficiently. The AI catches inconsistencies in documentation, suggests implications I haven't considered, and can read the entire repository in a way that I (as a human) can only partially keep in memory.

The repository structure is explicitly designed around this collaboration: the SSoT model reduces AI hallucination (contradictory facts in multiple files confuse AI agents), the handoff log provides context across sessions, and the ADRs prevent re-litigating settled decisions.

What This Taught Me

Enterprise practices exist for reasons. ADRs, risk registers, SLO definitions, and handoff logs aren't bureaucratic overhead — they're tools for operating complex systems reliably over time. A homelab is complex enough to benefit from all of them, even at small scale.

The practices also compound: a risk register is more useful when it references ADRs. A handoff log is more useful when it references the risk register. Documentation discipline isn't a separate "non-technical" activity — it's infrastructure for the engineering work itself.
