Infrastructure as Code: Provisioning VMs with OpenTofu

18 April 2026

Tags: homelab, opentofu, terraform, iac, proxmox, devops

I could have clicked through the Proxmox GUI to create VMs. It takes maybe 5 minutes per VM, it works, and for a homelab that's probably fine. I didn't do that. This post is about why writing HCL to declare four VMs across two Proxmox hosts is worth the extra setup time — and one very honest admission about a known gap in my approach.

Why OpenTofu Over Terraform

Terraform is the industry standard for Infrastructure as Code. I'd used it at work. The obvious move was to use Terraform here.

I chose OpenTofu instead. OpenTofu is an open-source fork of Terraform created after HashiCorp changed Terraform's license from MPL to BSL in 2023. The BSL license restricts certain competitive uses in ways the MPL didn't, and the infrastructure-as-code community responded by forking the project under the OpenTofu project (now part of the Linux Foundation).

For a homelab, the practical differences are minor — OpenTofu is fully compatible with Terraform providers and most Terraform configurations work without modification. The reason I went with OpenTofu was partly philosophical (I want my tools to be genuinely open source) and partly practical (the OpenTofu community is active and the project's governance model is more transparent about future direction).

The bpg/proxmox Provider

The bpg/proxmox provider is the mature, actively-maintained OpenTofu/Terraform provider for Proxmox. It lets you declare Proxmox VMs in HCL and have OpenTofu make them real.

Authentication is via a Proxmox API token — not a username and password. The API token approach is better for several reasons: you can scope it to specific privileges, you can revoke it without changing a password, and you can generate one per tool or purpose. For OpenTofu, I created a dedicated service user (tofu@pve) and generated a token for it with only the privileges needed to manage VMs.

pveum user add tofu@pve
pveum role add TofuRole -privs "VM.Allocate VM.Clone VM.Config.CDROM \
  VM.Config.CPU VM.Config.Disk VM.Config.HWType VM.Config.Memory \
  VM.Config.Network VM.Config.Options VM.Audit VM.PowerMgmt \
  Datastore.AllocateSpace Datastore.Audit Sys.Audit"
pveum aclmod / -user tofu@pve -role TofuRole
pveum user token add tofu@pve tofu --privsep=0

The token value is shown once at generation — copy it immediately into terraform.tfvars, which is gitignored. The token ID (tofu@pve!tofu) goes in config files. The token secret never touches the repository.

Declaring the VMs

The actual HCL declares each VM with its target Proxmox host, CPU count, memory, disk size, and network config. For four VMs across two hosts, the configuration is maybe 100 lines of HCL — verbose by some standards, but explicit in a way that a GUI wizard isn't.

The workflow:

cd infra/
tofu init           # download providers
tofu plan           # show what will change
tofu apply          # make it so

tofu plan is the most useful step. It shows you exactly what OpenTofu is going to do before it does it — create, update, or destroy. For infrastructure, having a diff before you commit is invaluable. I've caught several configuration mistakes at the plan stage that would have been painful to fix after apply.

One practical detail: my first k3s server VM (k3s-server-01) existed in Proxmox before I wrote the OpenTofu configuration, because I'd set it up manually initially. OpenTofu's import command lets you pull an existing resource into the state file without recreating it:

tofu import proxmox_virtual_environment_vm.k3s_server_01 pve01/301

After import, tofu plan showed the drift between the actual VM config and what I'd declared in HCL. I reconciled the HCL to match reality, then applied — and from that point, OpenTofu owned the VM.

The IaC Workstation-Only Policy

I made an explicit decision (ADR-013) to run all IaC from my DevOps workstation only — not from a shared CI/CD system, not from a server in the cluster. The reasons:

Secrets management. Running tofu apply requires the Proxmox API token. I didn't want to manage that secret in GitHub Actions or a CI system until I had a proper secrets management strategy. The workstation is the one machine I control fully.

Blast radius. A misconfigured CI/CD pipeline that automatically applies infrastructure changes can destroy things faster than you can stop it. For a learning environment, I'd rather have the human in the loop on every apply.

This is a known trade-off. Manual execution means there's no automated check that the repo state matches the actual infrastructure. I've documented that gap in the risk register as R-006 — it'll be addressed in Phase 5 with GitHub Actions running tofu plan on PRs and requiring manual approval for apply.

The Known Gap: Local State File

The biggest honesty moment in this entire blog series: my OpenTofu state file is stored locally on my workstation.

The terraform.tfstate file is what OpenTofu uses to track what it's created and how it maps to the HCL declarations. If that file is lost or corrupted, OpenTofu loses track of the infrastructure it manages. You can usually recover by re-importing resources, but it's painful and error-prone.

The right solution is a remote state backend — a central store that the state file lives in, accessible from any machine, with locking to prevent concurrent applies. In a cloud environment you'd use an S3 bucket. In a self-hosted environment, MinIO provides an S3-compatible API.

I haven't deployed MinIO yet. It's the highest-priority item on the Phase 4 backlog (risk register: R-002, Critical). The state file is gitignored and backed up to the Synology NAS, but a local file on a single workstation is not a sustainable architecture. I'm documenting this explicitly because hiding known gaps in a portfolio piece is worse than admitting they exist.

What This Taught Me

IaC is most valuable for its documentation as much as its automation. The HCL files are a precise, executable specification of the infrastructure — every VM, its CPU count, its memory, which host it lives on, which network it's attached to. That's more accurate than any written documentation, because it's the thing that actually makes the infrastructure exist. When I look at the HCL, I know exactly what's deployed. When I look at a written doc, I wonder if it's still accurate.

The import-first workflow (import existing resources, reconcile drift, then manage going forward) is also a realistic pattern. In the real world, you often inherit infrastructure that wasn't created with IaC. The pattern of "import, drift-check, adopt" is how you bring that infrastructure under IaC management without recreating it.
